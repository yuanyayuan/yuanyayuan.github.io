Logstash数据同步

介绍

可以从从数据库中采集数据到es中，可以通过设置自增id主键或者时间来控制数据的自动同步。

  • id：假设有1000条数据，Logstash识别后会进行一次同步，同步完会记录这个id为1000，以后数据库新增数据，那么id会一直累加，Logstash会有定时任务，发现id大于1000了，则增量到es
  • 时间：假设有1000条数据，每条数据都有一个字段为time，初次同步完毕后记录这个time，下次同步的时候进行时间对比，如果有超过这个事件的，那么可以做同步。这里可以同步增加数据或者修改数据，因为同一条数据的时间更改会被识别，而id不会

  • 插件logstash-input-jdbc(已经集成在Logstash中)

  • 创建索引

  • JDK

ES集群的概念

Cluster（集群）：群集是一个或多个节点的集合，它们一起保存整个数据，并提供跨所有节点的联合索引和搜索功能。每个群集都有自己的唯一群集名称，节点通过名称加入群集。

  • 索引分片相当于蛋糕被分成好几块
  • 副本分片是主分片的备份，主挂了，备份还是可以访问
  • 同一个分片的主与副本是不会放在同一个服务器里的。

集群脑裂

什么是脑裂

发生网络中断或者服务器宕机，那么集群会有可能被划分成两个部分，各自有自己的master管理

解决方案

master主节点要经历过多个master节点共同选举后才能成为新的主节点，需要半数以上的同意。

原理：半数以上的节点同意选举，节点才能成为新的master

discovery.zen.minimum_master_nodes=(n/2)+1

  • n为集群中master节点的数量（node.master=true设置的那些服务器节点总数）
  • ES7.X中移除minimum_master_nodes，完全由ES自己去控制

集群文档读写原理

文档写原理

  1. 客户端会选择某一个节点作为协调节点，负责接收用户请求
  2. 协调节点会根据文档进行路由，数据要写到哪一个主分片里面（根据文档id进行hash算法）
  3. 数据同步至副本分片
  4. 请求返回协调节点
  5. 告知客户端

文档读原理

  1. 客户端会选择某一个节点作为协调节点，负责接收用户请求
  2. 协调节点根据文档id进行hash算法，计算出文档在那个位置，读数据是从主分片和副本分片轮询读取
  3. 请求返回协调节点
  4. 告知客户端

中间件-Elasticsearch深度分页与批量操作

发表于 2021-05-09 | 分类于 elasticsearch - elasticsearch深度分页与批量操作

ES分页

分页查询

搜索分深浅，比如第一页，第二页，第十页是比较浅的分页，第200000页就比较深了

深度分页

在获取9999条到10009条时，其实每个分片都会拿到10009条数据，延后集合在一起，总共10009*3=30027条数据，针对这30027条数据在做排序处理，最终获得最后10条数据。

搜索的太深就会带来新能问题，消耗CPU和内存。es为了性能不支持一万条数据以上的分页。

为了避免此问题，避免深度分页操作，比如限制分页页数，只支持100页展示。

解除分页限制

企业内部有些业务需突破10000条限制

滚动分页

一次性查询1万+的数据，会影响性能，可以采用滚动搜索。滚动搜索可以先查询出一些数据，然后在紧接着依次往下查询。

原理

第一次查询的时候会有一个滚动ID，相当于一个锚标记，随后再次滚动搜索时需要上一次的锚标记，根据这个进行下一次搜索请求。每次搜索都是基于一个历史的数据快照，查询数据期间，如果有数据变更，那么和搜索是没有关系的，搜索内容还是快照内的内容

  • 第一次搜索
  • 第二次搜索

批量操作bulk

基本语法

  • { action: { metadata }}代表批量操作的类型，可以是新增、删除或修改
  • \n是每行结尾必须填写的一个规范，每一行包括最后一行都要写，用于es的解析
  • { request body }是请求body，增加和修改操作需要，删除操作则不需要

批量操作的类型

action 必须是以下选项之一:

  • create：如果文档不存在，那么就创建它。存在会报错。发生异常报错不会影响其他操作。
  • index：创建一个新文档或者替换一个现有的文档。
  • update：部分更新一个文档。
  • delete：删除一个文档。

metadata中需要指定要操作的文档的_index、_type和_id,_index、_type也可以在url中指定

实操

  • create新增文档数据，在metadata中指定index以及type

  • create创建已有id文档，在url中指定index和type

  • index创建，已有文档id会被覆盖，不存在的id则新增
  • update跟新部分文档数据
  • delete批量删除
  • 综合批量各种操作

例如搜索address字段中包含mill的文档，对比搜索数值可以发现，对于数值类型match操作使用的是精确匹配，对于文本类型使用的是模糊匹配；

短语匹配搜索（match_phrase）

使用match_phrase表示，例如搜索address字段中同时包含mill和lane的文档：

  • slop: 允许词语间跳过的数量

match拓展（operator，minimum_should_match）

  • operator
    • or: 搜索内容分词后，只要存在一个词语匹配就展示结果
    • and: 搜索内容分词后，都要满足词语匹配
  • minimum_should_match

最低匹配精度，至少有[分词后的词语个数]x百分百，得出一个数据值取整。举个例子：当前属性设置为70，若一个用户查询检索内容分词后有10个词语，那么匹配度按照 10x70%=7，则desc中至少需要有7个词语匹配，就展示；若分词后有8个，则 8x70%=5.6，则desc中至少需要有5个词语匹配，就展示。

根据文件主键ids搜索

在多字段中查询（multi_match），字段设置权重（boost）

  • multi_match

满足使用match在多个字段中进行查询的需求

  • boost

权重，为某个字段设置权重，权重越高，文档相关性得分就越高。通畅来说搜索商品名称要比商品简介的权重更高。

email^10 代表搜索提升10倍相关性，也就是说用户搜索的时候其实以这个email为主，firstname为辅，email的匹配相关度当然要提高权重比例了。

组合查询

  • 组合搜索，使用bool来进行组合，must表示同时满足，例如搜索address字段中同时包含mill和lane的文档；
  • 组合搜索，should表示满足其中任意一个，搜索address字段中包含mill或者lane的文档；
  • 组合搜索，must_not表示同时不满足，例如搜索address字段中不包含mill且不包含lane的文档；
  • 组合搜索，组合must和must_not，例如搜索age字段等于40且state字段不包含ID的文档；

过滤搜索

对搜索出来的结果进行数据过滤。不会到es库里去搜，不会去计算文档的相关度分数，所以过滤的性能会比较高，过滤器可以和全文搜索结合在一起使用。
post_filter元素是一个顶层元素，只会对搜索结果进行过滤。不会计算数据的匹配度相关性分数，不会根据分数去排序，query则相反，会计算分数，也会按照分数去排序。

  • query：根据用户搜索条件检索匹配记录

  • post_filter：用于查询后，对结果数据的筛选

  • gte：大于等于

  • lte：小于等于

  • gt：大于

  • lt：小于

使用例子

  • 搜索过滤，使用filter来表示，例如过滤出balance字段在20000~30000的文档；

搜索聚合

  • 对搜索结果进行聚合，使用aggs来表示，类似于MySql中的group by，例如对state字段进行聚合，统计出相同state的文档数量；
  • 嵌套聚合，例如对state字段进行聚合，统计出相同state的文档数量，再统计出balance的平均值；
  • 对聚合搜索的结果进行排序，例如按balance的平均值降序排列；
  • 按字段值的范围进行分段聚合，例如分段范围为age字段的[20,30] [30,40] [40,50]，之后按gender统计文档个数和balance的平均值；
